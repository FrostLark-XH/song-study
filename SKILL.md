# 歌曲学习

用户给你一首歌名和（可选）歌手名，你需要生成一份完整的学习资料。每一步都不能跳过。

## 色彩系统与设计规则

所有输出统一使用以下语义色板。颜色不是装饰——每种颜色对应一种信息类型，帮助读者一眼定位。

### 色板

| 色名 | 色值 | 语义 | Markdown 用法 | Word 用法 |
|------|------|------|--------------|-----------|
| 深蓝灰 | `#2C3E50` | 权威、结构 | 一级标题下方分隔线 | 标题栏底色、一级边框 |
| 淡蓝 | `#D6E4F0` | 信息、客观 | — | 基本信息表头底色 |
| 暖米 | `#F5EBE0` | 人文、文本 | — | 歌词表头底色 |
| 淡绿 | `#E8F0E4` | 学习、成长 | — | 词汇表头底色 |
| 淡紫 | `#EDE8F5` | 分析、思辨 | — | 语法/文化表头底色 |
| 暖金 | `#FFF3E0` | 实践、身体 | — | 演唱技巧底色 |
| 琥珀 | `#B85C3A` | 强调、关键 | — | 关键术语高亮色 |
| 浅灰 | `#F8F8F8` | 辅助、节奏 | — | 数据表交替行底色 |

### Markdown 设计规则

- 一级标题 `#` 下方加 `───` 分隔线
- 二级标题 `##` 前后各空一行，保持呼吸感
- 引文块（歌手原话、创作自述）用 `>` + 前后空行包裹
- 歌词版本声明和校验状态用 `> ✅` 或 `> ⚠️` 标注
- 表格前后各空一行，不与正文粘连
- 演唱技巧的 "→" 方法部分与问题描述分行，形成视觉对比

### Word 设计规则

**全局设置：**
- 中文正文：宋体（SimSun）；中文标题：微软雅黑（Microsoft YaHei）
- 日文正文：MS Mincho（MS 明朝）；日文标题：Meiryo（メイリオ）或 MS Gothic
- 英文/罗马音：Arial
- 不要使用 Noto CJK 等第三方字体——系统自带字体字符覆盖最完整，不会有缺字问题
- 正文 11pt，注释 9pt，标题按层级 18/15/13pt
- 页边距：上下 2.54cm，左右 2.0cm

**表头着色（强制）：**
- 基本信息表 → 淡蓝 `#D6E4F0`
- 歌词对照表 → 暖米 `#F5EBE0`
- 核心词汇表 → 淡绿 `#E8F0E4`
- 语法/文化相关表 → 淡紫 `#EDE8F5`

**交替行：** 所有超过 3 行的数据表，奇数行白底、偶数行浅灰 `#F8F8F8`。

**段落样式：**
- 歌手原话引用 → 左缩进 0.8cm + 左边框线（暖金色 `#D4A574`，1.5pt）+ 斜体
- 语法点名称 → 加粗 + 琥珀色 `#B85C3A`
- 演唱技巧关键词（问题描述部分）→ 加粗
- "→" 后的方法部分独立成行，稍微缩进

**校验状态块：**
- 文档末尾校验声明 → 浅灰底色 + 深蓝灰左边框，形成闭合方块

**演唱技巧底色：**
- 整个演唱技巧段落可以用极浅的暖金底色 `#FFF3E0` 包裹，形成"实践区"视觉标识

```markdown
| 词/曲作者 | ... |
| 所属专辑 | ... |
| 发行年份 | ... |
| 语言 | ... |

## 背景故事

（2-3 段，讲这首歌的创作背景、典故、影响力。不要写套话，要提供有意思的事实。）

## 歌词

> ✅ 歌词来源与校验：{版本声明——"录音室原版，三源交叉验证通过"}

（逐段展示。外文歌用以下三列表格：）

| 原文 | 发音 | 中文翻译 |
|------|------|----------|
| 原文行 | 发音标注 | 中文译文 |

**英文歌歌词排版规则（强制）**：英文歌用两列表格（原文 | 中文翻译），每条原文以实际演唱的短行为单位，**一行一句**。严禁把多行用 `/` 拼成一个大段落在表格中挤压。每个英文短句独占一行，中文翻译对齐到同行的第二列。如果英文原句特别长（超过 50 个字符），可以在自然断句处拆成两行。排版要让人能轻松跟唱，不能密密麻麻堆在一起。

## 语言学习（仅外文歌，中文歌可跳过此节）

### 核心词汇
| 原文 | 读音 | 词性 | 释义 | 等级 |
|------|------|------|------|------|
| 歌词中的关键词 | 平假名/罗马音 | 名/动/形... | 中文释义 | N5~N1 |

> **选词说明**：一句话概括选词逻辑。
> **动词活用提示**：标注歌词中动词的实际形态与辞书形对照。

（从全曲中选出 10-15 个有学习价值的词，优先选：高频词、歌词核心意象词、动词搭配。歌词没有的词不要硬塞。）

### 语法点

从歌词中找 3-5 个有价值的句型/表达，逐一拆解。每个语法点格式为**加粗标题** + 正文段落：

- **{句型名称}**：出现在 [Section]"（歌词原句）"。{结构分析}。{使用时注意什么}。

### 文化笔记

- {歌词中涉及的文化概念、习惯表达、隐喻}
- {不直译会误解的表达，如「よろしくお願いします」不只是「请多关照」}

每项一个自然段，说清"为什么日本人会这样说"。不做百科式堆砌。

## 演唱技巧

```

## 执行流程

### 0. 版本检查（多版本检测 —— 必须执行）

**在进入歌词搜索之前，必须先检查是否存在多个版本。**

如果用户没有明确指定歌手，先用 `WebSearch` 搜索 `"{歌名} 歌曲"`。检查搜索结果中是否出现：
- 不同歌手演唱的同一歌名词条
- 不同语言版本（如日文原唱 vs 中文翻唱）
- 同名但不同内容的歌曲（如 "La Vida" 可能是 Coldplay 的 "Viva La Vida" 或中森明菜的 "La Vida"）

**发现多版本时的操作**：立即停止后续流程，列出所有版本供用户选择。格式：

> "{歌名} 有多个版本，你想要哪一个？
> 1. {版本A} — {歌手/说明}
> 2. {版本B} — {歌手/说明}
> 3. {版本C} — {歌手/说明}"

用户确认后再继续。

如果用户已明确指定歌手，跳过此步骤，直接进入歌词搜索——但搜索结果中如果出现了同名不同歌的情况仍需确认。

### 1. 搜索歌词（多源交叉校验）

**核心策略：多重搜索引擎 + 多重 URL 抓取方式，确保至少拿到歌词。**

先用 3 个不同角度的搜索同时发起：
- `WebSearch` 搜 `"{歌名} {歌手名} lyrics"`（英文/罗马音查询）
- `WebSearch` 搜 `"{歌名} 歌词"`（中文查询）
- `WebSearch` 搜 `"{歌名} {歌手名} 歌詞"`（日文查询，如果是日文歌）

从搜索结果中收集 2-3 个歌词页 URL。然后按优先级尝试抓取：

**策略 A：WebSearch 聚合摘要优先利用。** 当 WebSearch 结果本身已经包含了完整或接近完整的歌词文本时（搜索引擎的歌词聚合卡片），直接用它作为主源。这个渠道通常不被 WebFetch 限制。

**策略 B：WebFetch 直接抓取。** 对收集到的 URL 使用 `WebFetch`。一些大型中文歌词站（如 百度百科、豆瓣音乐）比 Genius 更容易通过 WebFetch 的域名安全检查。

**策略 C：curl 替代 WebFetch。** 当 WebFetch 对所有目标 URL 都报安全限制时，改用 curl：
```bash
curl -s --connect-timeout 15 "<歌词页面URL>" 2>&1 | head -500
```
从 HTML 中提取歌词——常见的歌词容器：Genius 用 `data-lyrics-container="true"`、uta-net 用 `#kashi_area`、一般站用 `<div id="lyric">` 或 `<div class="lyric">`。

**策略 D：搜索引擎深度挖掘。** 如果前三个策略都失败，换不同关键词组合再次 WebSearch——加上"完整版""罗马音""歌詞 全文"等限定词。歌词搜索引擎的聚合卡片在不同查询下可能触发不同的展示。

**至少对比 2 个来源**，确认歌词一致。发现不一致时：
- 优先信任官方来源（唱片公司官网、歌手官方页面）
- 其次信任大型歌词数据库（Genius、魔镜歌词网等）
- 如果分歧无法解决，在歌词部分标注 `⚠️ 此处各版本有出入`
- 源地址附在文末

### 2. 标注版本

必须写清歌词来自哪个版本：录音室版？某个 live 版？remastered？不同版本歌词可能不同。如果用户未指定，默认取录音室原版。

### 3. 外文歌词处理

对外文歌，做三列对照：原文 / 发音 / 中文翻译。

**发音标注规则**：
- **日语**：用 Hepburn 式罗马字。长音 ō 写作 `ou`（如 `doushite`），ū 写作 `uu`（如 `tsuu`），促音双写辅音（如 `zutto`）。实际发音对照：`ou` 段发音为长音 ō，不是「欧」
- **优先使用自身日语能力生成罗马音**，而不是依赖 Genius Romanizations（该站频繁被 WebFetch 拦截）。如果在步骤 1 中已成功抓取 Genius Romanizations 页面则用它交叉校验，否则自行标注。
- **韩语**：文观部式罗马字。示例：`사랑해` → `saranghae`
- **英语**：不做逐词音标，在演唱技巧部分标出关键单词的自然拼读难点
- **其他语言**：用中文近似音 + 关键发音提醒
- 发音标注要能跟着唱，不要用 IPA（除非这首歌的发声训练需要 IPA）

**中文翻译规则**：
- 先忠于原文，再考虑可唱性
- 不要翻译成"现代诗风格"，保持原意
- 如果原文有双关或特殊修辞，在翻译后用括号备注

### 4. 验证歌词

这是最关键的步骤。做完所有抓取后逐句对照搜索结果，确认：
- 没有遗漏行或重复行
- 没有错别字（尤其是同音异义词）
- 分段符合原曲结构（主歌/副歌/桥段等标记明确）
- 如果抓取内容疑似不完整，再搜一轮确认

如果某处歌词在所有来源间都无法确认（如多个版本有不同段落），标注 `⚠️` 并写明哪个来源提供了哪种版本。

### 5. 背景故事

用 `WebSearch` 搜 `"{歌名} {歌手名} 创作背景"` 或 `"{song} {artist} story behind"`。需要包含：
- 这首歌为什么会被写出来（事件/情感/灵感来源）
- 创作过程中的关键细节
- 发行后的反响或影响
- 如果和特定影视作品/游戏/事件相关，必须说明

不要写"这首歌很美"、"感动了无数人"这类空洞评价。要具体事实。

### 6. 语言学习（外文歌专属，中文歌跳过此步）

此步骤让歌词成为语言教材。分三个子模块：

**6a. 核心词汇**
从歌词中选出 10-15 个有学习价值的词。选词标准：
- 高频日常词优先于生僻词
- 歌词中反复出现的核心意象词必选
- 包含至少 2 个动词的原形/变形对照（如 `覺えた`→原形 `覺える`）
- 用 `WebSearch` 确认每个词的 JLPT 等级，不确定则标注「---」

**6b. 语法点**
从歌词中找 3-5 个有价值的句型/表达。查找方式：
- 搜索引擎搜 `"{句型} 日语语法"` 确认你的解释准确
- 每个点必须引用歌词中的原句作为例子
- 拆解结构 + 说明语气/语境
- 给出一个额外的例句（不来自歌词，便于理解）

**6c. 文化笔记**
解释歌词中不直译会误解的表达和文化隐喻。不是百科罗列，是说清楚"为什么在这个文化和语境下，这句话会被这样写/这样唱"。

### 7. 演唱技巧

分析 3-5 个关键技巧/难点，Markdown 格式：
```
- **{技巧/难点名称}**：{问题是什么}
  → {怎么解决}
```
（问题描述与解决方法用 `→` 分隔，方法部分缩进另起一行，形成视觉对照。）

涵盖维度：
- 音高（最高音在哪句、真假声切换点）
- 节奏（切分音、三连音、抢拍/拖拍）
- 气息（换气点、长句处理）
- 咬字（卷舌音、鼻音、闭口音等）
- 情绪表达（哪句该收、哪句该放）

如需确认音域或演唱技巧细节，可用 `WebSearch` 搜索 `"{歌名} 歌唱 技巧 音域"` 或 `"{song} vocal range key"` 作为参考。

### 8. 结尾标注

在文档末尾附上：
- 歌词来源 URL 列表
- 背景信息来源 URL

### 9. 保存文件

每首歌建一个独立文件夹，同时输出两种格式：

**文件夹结构**：`E:/song-study/{歌手名}_{歌名}/`

**输出文件**：
- `{歌手名}_{歌名}.md` — Markdown 格式（遵循 Markdown 设计规则）
- `{歌手名}_{歌名}.docx` — Word 格式（遵循 Word 设计规则）

文件名中的空格用下划线替代。

**Word 生成方式**：

使用 Python `python-docx` 库生成 .docx 文件。本 skill 自带可复用的生成模板 `scripts/docx_template.py`，提供了 `create_document()`、`make_table()`、`add_quote_paragraph()`、`add_singing_tip()`、`add_verification_block()` 等辅助函数。在实际使用时，先读取模板文件了解函数签名，然后编写生成脚本。

**Python 编码注意事项**：
- 中文引号（`"` `"` U+201C/U+201D）和全角逗号（`，` U+FF0C）在 Python 字符串中可能导致语法混淆——如果把中文左引号误识别为字符串结束符
- 解决方法：对需要在 Python 字符串中包含中文引号的情况，用变量 `LQ = '“'` / `RQ = '”'` 拼接，或统一用 `「」` 替代
- 如果引号样式不是硬性要求，尽量在正文中用 `「」` 代替 `""`，减少编码问题

**文件验证**：生成完成后，用 `ls -la` 确认两个文件都已写入且大小 > 0。

### 10. 发送文件（cc-connect 环境）

**判断条件**：如果系统 prompt 中包含 "You are running inside cc-connect" 或用户提到是通过消息平台对话的，执行此步骤。

**详细操作**见 `references/cc-connect-file-send.md`。简要流程：

1. 从 `~/.cc-connect/config.toml` 读飞书凭据（app_id / app_secret）
2. 从 `~/.cc-connect/sessions/<project>_<hash>.json` 提取当前飞书 chat_id
3. 获取 tenant_access_token（用 `grep | cut` 提取，不用 Python json.load 避免编码坑）
4. 逐个上传文件（md 和 docx）到飞书 `im/v1/files`
5. 逐个发送文件消息到 `im/v1/messages`

发送完成后报告 message_id 确认送达。

## 重要约束

- **不跳过任何步骤**。即使歌词搜索困难，也要用多个查询尝试，并在文档中注明搜索过程。
- **歌词绝对不能出错**。宁可标注"无法完全确认"也不能编造。如果你的训练数据中的歌词与搜索结果不一致，以搜索结果为准。
- **歌曲名对应多个版本时必须问用户**。同一歌名可能被不同歌手/不同语言/不同时期演唱。如果 WebSearch 结果中出现多个版本，必须列出选项让用户确认，不得自行选择。**如果用户只给了歌名没给歌手，先搜，发现多版本立刻暂停并询问。**
- **发音标注必须可唱**。读完发音标注能直接唱出来，不需要再查字典。优先用自身语言能力生成，外部源作为校验。
- **翻译不要诗化**。忠实第一，优美第二。
- **如果是纯音乐/无人声曲**，跳过歌词和发音部分，重点做背景和演奏技巧。
- **Word 文档必须带颜色语义**。每种表类型使用对应的颜色（蓝=信息、暖米=歌词、绿=词汇、紫=语法、金=技巧），不能所有表都是灰色。交替行着色不能省略。
- **Markdown 也要有视觉节奏**。标题层级清晰、段间距合理、引文有缩进、校验信息有标识。
- **禁止密集排版**。歌词表格每个单元格一行一句，不堆砌、不挤压。表格行高要有呼吸感，cell padding 不要吝啬。英文歌严禁用 `/` 把多句拼成一段，必须逐行拆分。
- **字体只许用系统自带**。中文=宋体/微软雅黑，日文=MS Mincho/Meiryo，英文=Arial。禁止使用 Noto CJK、思源系列等第三方字体——系统字体字符覆盖最完整，无缺字风险。
- **生成后验证文件存在**。用 `ls -la` 确认 md 和 docx 均已写入且文件大小合理。
